Tablas de hash y diccionarios

1 Objetivo de aprendizaje

Comprender que una tabla de hash es una forma de implementar un diccionario llave-valor. Al estudiar esta página deberías poder:

  • explicar qué es un diccionario;
  • distinguir llave, valor, índice y bucket;
  • definir una función de hash;
  • aplicar hashing modular;
  • reconocer y resolver colisiones mediante encadenamiento o direccionamiento abierto;
  • usar sondeo lineal;
  • calcular el factor de carga;
  • explicar cuándo hacer rehashing;
  • reconocer que una tabla de hash no mantiene orden.

2 Antes de partir: cuatro palabras clave

Antes de usar fórmulas, conviene separar cuatro ideas que suelen mezclarse:

  • Llave: dato usado para buscar.
  • Valor: información asociada a la llave.
  • Índice: posición numérica dentro del arreglo.
  • Bucket: celda de la tabla donde pueden caer una o más llaves.
Concepto Qué significa Ejemplo
Llave Dato usado para buscar rut_estudiante
Valor Información asociada datos del estudiante
Índice Posición en el arreglo A[3]
Bucket Celda de la tabla lista guardada en A[3]
Importante

Una tabla de hash está diseñada para buscar por llave, no para mantener las llaves ordenadas. No debe usarse para responder sucesor, predecesor, mínimo o máximo.

3 Idea central del tema

Un diccionario es una estructura que almacena pares:

\[ (\text{llave}, \text{valor}) \]

La llave identifica el dato y el valor es la información asociada. Por ejemplo:

RUT -> datos de estudiante
codigo_producto -> stock disponible
tipo_tramite -> cola de personas esperando

Una tabla de hash implementa este diccionario usando un arreglo A de tamaño m. Para decidir en qué posición guardar o buscar una llave k, se calcula un índice con una función de hash. La celda relevante se escribe:

\[ A[h(k)] \]

En palabras: la llave entra a la función h; la función devuelve un índice válido; ese índice indica qué bucket del arreglo se debe revisar.

Nota

La tabla no guarda los elementos según su orden natural. La posición depende de h(k) y de la estrategia usada para manejar colisiones.

4 Cuándo usar tablas de hash

Usa una tabla de hash cuando el problema pide acceso rápido a partir de una llave:

  • verificar si un elemento pertenece a un conjunto;
  • contar frecuencias;
  • asociar identificadores con datos;
  • agrupar elementos por categoría;
  • acceder a una lista, cola, pila, heap u otra estructura asociada a una clave;
  • implementar un diccionario con operaciones esperadas eficientes.

No es la estructura adecuada si el problema pide principalmente:

  • obtener el mínimo o máximo global;
  • recorrer las llaves en orden creciente;
  • encontrar sucesor o predecesor;
  • mantener un ranking ordenado;
  • responder consultas de rango.
Necesidad del problema Estructura más natural
Buscar por llave Tabla de hash
Obtener mínimo o máximo rápidamente Heap
Atender por orden de llegada Cola FIFO
Mantener llaves ordenadas Árbol binario de búsqueda balanceado
Modelar relaciones entre entidades Grafo
Saber si dos elementos están en la misma componente Union-Find
Advertencia

Hash y heap no son lo mismo. Un hash busca por llave; un heap entrega rápido el mínimo o máximo según prioridad. Si el problema pregunta “el más chico”, “el más grande” o “el siguiente”, una tabla de hash por sí sola no es suficiente.

5 Conceptos mínimos

5.1 Diccionario

Un diccionario almacena pares (llave, valor) y ofrece estas operaciones principales:

  • asociar un valor v a una llave k;
  • actualizar el valor asociado a una llave ya existente;
  • obtener el valor asociado a una llave;
  • eliminar una llave y su valor.

La llave debe permitir identificar el par buscado. Si hay colisiones, no basta guardar solo el valor: también se debe guardar la llave para verificar que el elemento encontrado corresponde a la búsqueda.

5.2 Función de hash

5.2.1 1. Intuición

Una función de hash es una regla que convierte una llave en un número que sirve como índice del arreglo.

Por ejemplo, si la llave es rut_estudiante, la tabla no puede usar directamente el RUT completo como posición física. Primero necesita transformarlo en un índice pequeño, como 0, 1, 2 o 3, dependiendo del tamaño de la tabla.

La función hash no guarda datos por sí misma. Solo calcula una posición tentativa. Después, la tabla usa esa posición para insertar, buscar, actualizar o eliminar.

5.2.2 2. Definición formal

Si K es el conjunto de llaves posibles y la tabla tiene tamaño m, la función hash se escribe:

\[ h : K \to \{0, \dots, m - 1\} \]

Esto significa que h recibe una llave y entrega un índice válido entre 0 y m - 1.

5.2.3 3. Explicación de símbolos

Donde:

  • K es el conjunto de llaves posibles.
  • m es el tamaño de la tabla.
  • h(k) es el índice calculado para la llave k.
  • n es la cantidad de elementos guardados.

El acceso al bucket calculado se representa así:

\[ A[h(k)] \]

Aquí A es el arreglo de la tabla y A[h(k)] es la celda que se revisa para la llave k.

5.2.4 4. Ejemplo con hashing modular

Un ejemplo típico para llaves enteras es el hashing modular:

\[ h(k) = k \bmod m \]

Si m = 7 y k = 15, entonces:

15 mod 7 = 1

Por lo tanto, la llave 15 apunta inicialmente al bucket A[1].

5.2.5 5. Advertencia sobre colisiones

La posición calculada es tentativa porque dos llaves distintas pueden producir el mismo índice. Si eso ocurre, la tabla necesita una estrategia de colisiones.

Advertencia

Calcular h(k) no garantiza que el bucket esté libre. Solo indica por dónde comenzar la búsqueda o inserción.

5.3 Tabla de hash

Una tabla de hash es un arreglo A indexado por el valor de la función de hash. Para una llave k, se revisa la posición:

\[ A[h(k)] \]

Como el espacio de llaves suele ser mucho más grande que el tamaño de la tabla, distintas llaves pueden caer en la misma celda.

5.4 Colisión

Hay una colisión cuando dos llaves distintas producen el mismo índice:

\[ h(k_1) = h(k_2), \quad k_1 \ne k_2 \]

Con m = 7:

  • h(37) = 37 mod 7 = 2
  • h(51) = 51 mod 7 = 2

Ambas llaves quieren usar A[2]. Eso es una colisión.

Con h(k) = k mod 7, también ocurre:

37 mod 7 = 2
51 mod 7 = 2
58 mod 7 = 2

Las llaves 37, 51 y 58 colisionan en el índice 2.

6 Operaciones principales

6.1 Asociar o insertar

Para asociar v a la llave k:

  1. Calcular h(k).
  2. Ir a la celda A[h(k)].
  3. Si la llave ya existe, actualizar su valor.
  4. Si la llave no existe, insertar el par (k, v).
  5. Si hay colisión, aplicar la estrategia correspondiente.

6.2 Actualizar

Actualizar consiste en buscar la llave y reemplazar solo su valor asociado. La llave se mantiene.

antes:  "ana" -> 3
despues: "ana" -> 4

6.3 Obtener

Para obtener el valor asociado a k, se calcula h(k) y se busca la llave en la celda o en la secuencia de celdas correspondiente.

6.4 Eliminar

La eliminación depende de la estrategia de colisiones:

  • con encadenamiento, se elimina el nodo desde la lista del bucket;
  • con direccionamiento abierto, eliminar es más delicado, porque reemplazar una celda por NIL puede cortar búsquedas futuras.

7 Pseudocódigo claro

7.1 Hash directo ideal

El siguiente pseudocódigo directo solo es correcto en el escenario ideal sin colisiones o cuando la celda A[h(k)] maneja internamente las colisiones.

Input: tabla A, llave k.
Output: valor asociado a k, si existe.

HashSearch(A, k):
    indice <- h(k)
    return A[indice]

Input: tabla A, llave k, valor v.
Output: tabla A con v asociado a k.

HashInsert(A, k, v):
    indice <- h(k)
    A[indice] <- v

Input: tabla A, llave k.
Output: tabla A sin el valor asociado a k.

HashDelete(A, k):
    indice <- h(k)
    A[indice] <- NIL
Advertencia

Este modelo directo sirve para entender la idea base, pero no basta en una tabla real con colisiones. En una implementación real se guarda el par (llave, valor), no solo el valor.

7.2 Encadenamiento

En encadenamiento, cada celda del arreglo contiene una lista de pares (llave, valor). En vez de guardar un solo par en la celda, la celda apunta a una lista.

Input: tabla A, llave k.
Output: valor asociado a k, si existe.

ChainedHashSearch(A, k):
    indice <- h(k)

    for cada par en A[indice]:
        if par.llave = k:
            return par.valor

    return NOT_FOUND

Input: tabla A, llave k, valor v.
Output: tabla A con (k, v) insertado o actualizado.

ChainedHashInsert(A, k, v):
    indice <- h(k)

    for cada par en A[indice]:
        if par.llave = k:
            par.valor <- v
            return

    insertar (k, v) al inicio de A[indice]

Input: tabla A, llave k.
Output: tabla A sin el par cuya llave es k.

ChainedHashDelete(A, k):
    indice <- h(k)
    eliminar desde A[indice] el par cuya llave es k

7.3 Sondeo lineal

El sondeo lineal es una forma de direccionamiento abierto. Los elementos se guardan directamente en el arreglo. Si la posición h(k) está ocupada, se revisa la siguiente, luego la siguiente, y así sucesivamente:

\[ h(k),\ (h(k)+1) \bmod m,\ (h(k)+2) \bmod m,\ \dots \]

Los índices se calculan módulo m para volver al inicio de la tabla.

La variable pasos evita que el algoritmo dé vueltas infinitas si la tabla está llena o si la llave no aparece.

Input: tabla A, llave k, tamaño de tabla m.
Output: valor asociado a k, si existe.

LinearProbingSearch(A, k, m):
    indice <- h(k)
    pasos <- 0

    while A[indice] != NIL and pasos < m:
        if A[indice].llave = k:
            return A[indice].valor

        indice <- (indice + 1) mod m
        pasos <- pasos + 1

    return NOT_FOUND

Input: tabla A, llave k, valor v, tamaño de tabla m.
Output: tabla A con (k, v) insertado o actualizado.

LinearProbingInsert(A, k, v, m):
    indice <- h(k)
    pasos <- 0

    while A[indice] != NIL and pasos < m:
        if A[indice].llave = k:
            A[indice].valor <- v
            return

        indice <- (indice + 1) mod m
        pasos <- pasos + 1

    if pasos = m:
        error "tabla llena"

    A[indice] <- (k, v)

Input: tabla A, llave k, tamaño de tabla m.
Output: tabla A con la llave k marcada como eliminada, si aparece.

LinearProbingDelete(A, k, m):
    indice <- h(k)
    pasos <- 0

    while A[indice] != NIL and pasos < m:
        if A[indice].llave = k:
            A[indice] <- DELETED
            return

        indice <- (indice + 1) mod m
        pasos <- pasos + 1

    return NOT_FOUND
Advertencia

En sondeo lineal, la búsqueda debe seguir la misma secuencia usada por la inserción. No basta revisar solo A[h(k)].

8 Complejidad

Operación Complejidad esperada Comentario
Insertar/asociar O(1) Puede requerir manejar colisiones
Actualizar O(1) Primero se busca la llave
Obtener O(1) Esperada, no garantizada
Eliminar O(1) Más simple con encadenamiento
Rehashing O(n) Reinserta los n elementos
Nota

La complejidad O(1) es esperada. Puede degradarse si la función de hash distribuye mal las llaves o si el factor de carga es alto.

En el peor caso, muchas llaves pueden caer en el mismo lugar. Entonces una búsqueda puede terminar recorriendo muchos elementos, llegando a O(n).

9 Colisiones

Antes de elegir una estrategia, mira el problema mínimo:

Con m = 7:

  • h(37) = 37 mod 7 = 2
  • h(51) = 51 mod 7 = 2

Ambas llaves quieren usar A[2]. Eso es una colisión.

Hay dos respuestas clásicas:

  • Encadenamiento: en vez de guardar un solo par en la celda, la celda apunta a una lista.
  • Direccionamiento abierto: en vez de crear una lista, se busca otra posición dentro del mismo arreglo.

9.1 Encadenamiento

En encadenamiento, cada celda del arreglo es un bucket que contiene una lista de pares. Todas las llaves que llegan al mismo índice se guardan en esa lista.

Ejemplo con m = 7, h(k) = k mod 7 y llaves 15, 37, 51, 29, 58, insertando al inicio de cada lista:

Índice Lista
0 ∅
1 29 → 15
2 58 → 51 → 37
3 ∅
4 ∅
5 ∅
6 ∅

Ventajas:

  • eliminar es directo, porque se borra un nodo de la lista;
  • cada bucket puede almacenar más de un elemento;
  • es simple de razonar en ejercicios.

Desventajas:

  • si muchas llaves caen en el mismo bucket, la lista crece;
  • requiere estructuras adicionales para las listas.

9.2 Direccionamiento abierto

En direccionamiento abierto, todos los pares se almacenan dentro del mismo arreglo A. Si una celda está ocupada, se busca otra celda disponible siguiendo una regla de sondeo.

En sondeo lineal, la regla es revisar posiciones consecutivas. Para las mismas llaves del ejemplo, el resultado es:

Índice Valor
0 ∅
1 15
2 37
3 51
4 29
5 58
6 ∅

Ventajas:

  • no usa listas externas;
  • todo queda almacenado dentro del arreglo.

Desventajas:

  • se forman bloques de posiciones ocupadas;
  • al llenarse la tabla, las búsquedas se vuelven más largas;
  • eliminar requiere cuidado, normalmente usando marcas especiales de borrado.
Tip

Si el problema incluye muchas eliminaciones, encadenamiento suele ser más cómodo que direccionamiento abierto con sondeo lineal.

10 Factor de carga y rehashing

El factor de carga mide qué tan llena está la tabla:

\[ \lambda = \frac{n}{m} \]

Donde:

  • n es la cantidad de elementos guardados.
  • m es el tamaño de la tabla.

Mientras mayor es lambda, más probable es tener colisiones y secuencias largas de búsqueda.

10.1 Rehashing

Cuando el factor de carga supera un umbral razonable, se puede hacer rehashing:

  1. Crear una nueva tabla, normalmente más grande.
  2. Definir una nueva función hash, ajustada al nuevo tamaño.
  3. Reinsertar todos los elementos en la nueva tabla.

El costo del rehashing es O(n) porque hay que procesar los n elementos almacenados.

Importante

El rehashing es costoso cuando ocurre, pero permite recuperar búsquedas e inserciones esperadas O(1) al reducir colisiones.

11 Cómo leer el ejemplo guiado

El ejemplo se hará en tres etapas para no mezclar estrategias:

  1. Calcular el índice de cada llave.
  2. Insertar con encadenamiento.
  3. Insertar con sondeo lineal.

Primero todos usan la misma función h(k). Después se comparan dos formas distintas de resolver las colisiones.

12 Ejemplo guiado

Supongamos:

m = 7
h(k) = k mod 7
llaves = 15, 37, 51, 29, 58

Primero calculamos los índices:

Llave Cálculo Índice Qué ocurre
15 15 mod 7 = 1 1 Se inserta directo
37 37 mod 7 = 2 2 Se inserta directo
51 51 mod 7 = 2 2 Colisiona con 37
29 29 mod 7 = 1 1 Colisiona con 15
58 58 mod 7 = 2 2 Colisiona con 37 y 51

12.1 Etapa 1: cálculos

La tabla anterior solo calcula el índice inicial. Todavía no decide dónde queda cada llave si hay colisión.

12.2 Etapa 2: resultado con encadenamiento

Si cada nueva llave se inserta al inicio de la lista:

Índice Lista
0 ∅
1 29 → 15
2 58 → 51 → 37
3 ∅
4 ∅
5 ∅
6 ∅

Para buscar 58, se calcula h(58) = 2 y se revisa la lista del índice 2.

12.3 Etapa 3: resultado con sondeo lineal

En sondeo lineal, una colisión no crea una lista. Se sigue avanzando en el mismo arreglo hasta encontrar una celda libre.

Llave Índice inicial Revisión de posiciones Posición final
15 1 1 libre 1
37 2 2 libre 2
51 2 2 ocupado → 3 libre 3
29 1 1 ocupado → 2 ocupado → 3 ocupado → 4 libre 4
58 2 2 ocupado → 3 ocupado → 4 ocupado → 5 libre 5

Estado final:

Índice Valor
0 ∅
1 15
2 37
3 51
4 29
5 58
6 ∅

Para buscar 29, se calcula h(29) = 1. La búsqueda debe revisar:

1 -> 2 -> 3 -> 4

hasta encontrar la llave. Si una llave no está en la tabla, la búsqueda continúa hasta encontrar NIL o hasta completar m pasos.

13 Errores típicos

AdvertenciaError típico: intentar ordenar una tabla de hash

Una tabla de hash no mantiene orden. La posición física depende de h(k) y de las colisiones, no de la relación menor/mayor entre llaves.

AdvertenciaError típico: responder mínimo, máximo, sucesor o predecesor

La tabla de hash no entrega esas operaciones de forma eficiente. Para orden, usa una estructura ordenada; para mínimo o máximo frecuente, considera un heap.

AdvertenciaError típico: confundir hash con heap

Hash sirve para acceso por llave. Heap sirve para obtener mínimo o máximo según prioridad. Sus objetivos son distintos.

AdvertenciaError típico: guardar solo el valor y no la llave

Si hay colisiones, dos llaves distintas pueden llegar al mismo lugar. Por eso cada elemento debe conservar su llave junto con su valor.

AdvertenciaError típico: eliminar mal en direccionamiento abierto

Si se reemplaza una celda por NIL, se puede cortar la secuencia de búsqueda y hacer que otros elementos parezcan ausentes. Por eso se suelen usar marcas como DELETED.

AdvertenciaError típico: asumir que O(1) está garantizado siempre

La eficiencia esperada depende de una buena distribución, pocas colisiones y un factor de carga controlado. En el peor caso, una operación puede costar O(n).

14 Problema representativo de interrogación

Ver enunciado

Un sistema municipal recibe solicitudes para distintos tipos de trámite. Cada solicitud llega como:

(tipo_tramite, nombre_persona)

Ejemplos de tipo_tramite son "licencia", "certificado" y "permiso".

Cuando un módulo queda disponible, informa el tipo de trámite que puede atender. El sistema debe llamar a la persona que llegó primero para ese tipo de trámite y eliminarla de la espera.

Propón una estructura de datos eficiente y justifica:

  1. qué estructura principal usarías;
  2. qué se guarda como llave;
  3. qué se guarda como valor;
  4. cómo aseguras que se atienda primero a quien llegó antes para cada tipo;
  5. la complejidad esperada de insertar una solicitud;
  6. la complejidad esperada de atender una solicitud;
  7. por qué no corresponde usar solo un heap.
Ver solución

Primero identificamos qué exige el enunciado:

  • Acceso por tipo de trámite.
  • Orden de llegada dentro de cada tipo.
  • Eliminación del primero de la fila.

Por eso conviene usar una tabla de hash donde:

  • la llave es tipo_tramite;
  • el valor asociado a cada llave es una cola FIFO con las personas esperando ese trámite.

La estructura queda:

Hash[tipo_tramite] -> Cola FIFO de personas

Para agregar una solicitud:

AgregarSolicitud(H, tipo_tramite, nombre):
    if H no contiene tipo_tramite:
        H[tipo_tramite] <- cola vacia

    Enqueue(H[tipo_tramite], nombre)

Para atender:

AtenderSolicitud(H, tipo_tramite):
    if H no contiene tipo_tramite:
        return "no hay solicitudes"

    if H[tipo_tramite] esta vacia:
        return "no hay solicitudes"

    return Dequeue(H[tipo_tramite])

La tabla de hash permite acceder al grupo correcto en tiempo esperado O(1). La cola FIFO asegura que, dentro de cada tipo de trámite, se atienda a quien llegó primero.

Complejidades esperadas:

  • insertar solicitud: O(1) por acceso al hash más O(1) por Enqueue;
  • atender solicitud: O(1) por acceso al hash más O(1) por Dequeue.

No corresponde usar solo un heap porque el criterio no es obtener el mínimo o máximo global. La consulta siempre viene dada por un tipo específico de trámite y exige orden de llegada dentro de ese tipo. Eso se modela naturalmente con hash + cola FIFO.

16.2 Ejercicio 2
